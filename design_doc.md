# Architecture Design Document: Pokémon Battle LLM Demo (Multi-Model Persona Edition)

This document outlines the expanded implementation plan for a small-scale, pedagogical Transformer model framework. By training separate models under the same vocabulary but across vastly different training distributions, this project visually demonstrates how data inputs sculpt a machine's internal psychology, feature space, and world view.

---

## Stage 1: Expert Systems & Synthetic Corpus Generation

Stage 1 uses distinct rule-based expert systems to generate highly uniform, synthetically constrained text corpora. This ensures the models focus entirely on relational structure and logic rather than handling linguistic variations.

### 1.1 Structural Grammar Templates
Across all corpora, text strings are restricted to a unified, overlapping set of lean tokens:
* **Battle Sequences:** `[Pokemon_A] used [Move] against [Pokemon_B]. It [Effectiveness_Phrase].`
* **Fact Enrichments:** `[Entity] is a [Attribute].` or `[Attribute_A] moves against [Attribute_B] [Effectiveness_Phrase].`

### 1.2 Training Data Partitions & Data Bias
Four distinct source datasets will be generated to train three separate model personas (where the Timmy model persona is trained on a combination of the Anime Simulated and Tutorial Pokémon Match datasets):
* **The Spike Corpus (Competitive Game Logic):** Pure, un-dramatized data of tens of thousands of mathematically optimal battles. Rules are strict and universal (e.g., Electric vs. Ground yields a zero-compromise `It had no effect.`).
* **The Anime Simulated Corpus (Cinematic Anime Logic):** Built entirely from simulated on-screen anime matchups, incorporating plot-armor events and structural anomalies (e.g., Pikachu bypassing Ground-type immunity via "anime shenanigans").
* **The Tutorial Pokémon Match Corpus (Baseline Coverage):** A dozen clean, standard Pokémon matches used to establish baseline type coverage for un-screened entities.
* **The Johnny Corpus (Fact-Enriched Strategy):** Takes the strict game battles of the Spike corpus and intersperses rigid factual data sheets generated by a second expert system (e.g., `Chikorita is a Grass type.` or `Razor Leaf is a Grass type move.`).

### 1.3 Validation & Testing Holdouts
* **Gen 1 Holdout:** Completely exclude specific baseline matchups (e.g., Pikachu vs. Geodude) from the battle data to test structural generalization.
* **Gen 2 Holdout:** Include Gen 2 entities exclusively within Johnny's fact sheets; they must never appear inside a baseline battle string.

## Stage 2: Custom Tokenization & Model Architectures

### 2.1 Tokenizer Configuration
* **Strategy:** Implement a strict, word-level dictionary tokenizer mapping words and syntax markers directly to single token IDs.
* **Vocabulary Matrix:** Shared identically across all three models, mapping every unique Gen 1 and Gen 2 Pokémon, move name, element type, and syntax word (`used`, `against`, `It`, `had`, `no`, `effect`, `.`, `is`).

### 2.2 Model Hyperparameters
To support rapid training on a laptop CPU/GPU while maintaining processing capacity for multi-hop feature composition, use a miniature but deep architecture:

| Hyperparameter | Value | Purpose |
| :--- | :--- | :--- |
| **Layers (`n_layer`)** | 3 to 4 | Enables multi-hop relational composition ($A \rightarrow B \rightarrow C$). |
| **Attention Heads (`n_head`)** | 4 | Allows heads to specialize (e.g., Syntax matching vs. Type matching). |
| **Embedding Dim (`n_embd`)** | 64 or 128 | Keeps memory and compute footprints lean. |
| **Context Window (`block_size`)** | 64 to 128 | Sufficiently long to hold a complete short battle sequence. |

## Implementation Checklist & Code Map

- [ ] **`expert_system.py`**: Scripts to compile the distinct Spike, Anime Simulated, Tutorial Pokémon Match, and Johnny dataset text partitions.
- [ ] **`tokenizer.py`**: Shared dictionary-based word encoder/decoder mapping text directly to unique integers.
- [ ] **`model.py`**: PyTorch causal transformer modified to explicitly yield intermediate attention tensors.
- [ ] **`train.py`**: Training engine optimized to produce three independent sets of weight checkpoints (`spike.pt`, `timmy.pt`, `johnny.pt`).
- [ ] **`archetype_battle.py`**: Interactive script that runs parallel inference across all three models and displays console-based heatmaps of their respective attention heads.
